Core Cryptographic Concepts
🔐
01 / Symmetric Encryption
Same Key,
Both Ways
One shared secret key encrypts and decrypts. Fast and efficient for large data, but both parties must securely exchange the key first.
AES-128/192/256 are the modern standard (NIST FIPS 197, 2001)
DES (1977) used a 56-bit key — cracked by 1998
100 users need ~4,950 unique keys — doesn't scale
Block ciphers (AES) process fixed 128-bit chunks
Used in: File encryption · VPNs · HTTPS bulk data
🗝️
02 / Asymmetric Encryption
Public + Private
Key Pair
Each party holds a public key (shareable) and a private key (secret). A message encrypted with one can only be decrypted with the other.
RSA described publicly in 1977 by Rivest, Shamir & Adleman (MIT)
Security relies on the difficulty of factoring large prime numbers
Enables: confidentiality, integrity, authenticity, non-repudiation
Slower than symmetric — used for key exchange, not bulk data
Used in: Digital signatures · Email encryption · TLS handshake
🤝
03 / Diffie-Hellman Key Exchange
Shared Secret
in Public
Two parties agree on a shared secret over an open channel without ever transmitting the secret itself — using modular exponentiation math.
Published by Diffie & Hellman in 1976 — "New Directions in Cryptography"
Security relies on the discrete logarithm problem
Alone, it's vulnerable to Man-in-the-Middle (MitM) attacks
Combined with PKI certificates to defeat MitM
Used in: HTTPS · SSH · IPsec · Signal Protocol
#️⃣
04 / Cryptographic Hashing
One-Way
Fingerprint
Takes any input and produces a fixed-size digest. One-way (irreversible) and collision-resistant. Even a 1-bit change produces a completely different hash.
SHA-256 produces a 256-bit (64 hex char) digest regardless of input size
MD5 and SHA-1 are cryptographically broken — do not use
Passwords are stored as salted hashes, not plaintext
HMAC adds a secret key to a hash for message authentication
Used in: Password storage · File integrity · Digital signatures

▸ Agreeing on a Secret Key Over a Public Channel

Public Key Infrastructure (PKI)
▸ The Certificate Trust Chain
🌐
Your Browser
Connects to example.com over HTTPS. Requests the server's TLS certificate to verify its identity before exchanging data.
📄
Server Certificate
Contains the server's public key and identity. Digitally signed by an Intermediate CA. Browser checks the signature using the CA's public key.
🏛️
Intermediate Certificate Authority (CA)
Signs server certificates after verifying the requester's identity. Is itself signed by the Root CA. E.g., DigiCert, Let's Encrypt.
🔒
Root Certificate Authority
The ultimate trust anchor. Self-signed and pre-installed in your OS and browser. Everything traces trust back here. Kept offline for maximum security.
How It All Works Together — HTTPS Connection
▸ Every Padlock in Your Browser Follows This Flow
01
Certificate Request
Browser asks server for its TLS certificate and public key.
02
PKI Verification
Browser verifies the certificate signature against a trusted Root CA.
03
TLS Handshake
Asymmetric crypto (RSA / DH) negotiates a shared symmetric session key.
04
Symmetric Session
All data is now encrypted with AES using the agreed session key. Fast and secure.
05
Hashed + Signed
Each message is hashed and signed to guarantee integrity and authenticity.
